Open Source StreamYard on Your Own Stack

A browser-based live production studio using CLASP for real-time state and WebRTC signaling, PixiJS/vue3-pixi for GPU-accelerated compositing, and LVQR for broadcast output. No headless Chrome servers. No third-party infrastructure. Your protocols, your code, top to bottom.

The Core Insight

StreamYard composites video on the server using headless Chrome (3-4 CPU cores per broadcast). We composite on the host's GPU using PixiJS, which you already know inside out from LumenCanvas. The composited canvas gets captured via captureStream(), encoded with WebCodecs, and piped into LVQR's WebSocket ingest, which is already proven to work in the LVQR test-app.

CLASP replaces both the signaling server AND the state management layer. The AudioSpace/Clubhouse clone already demonstrates CLASP carrying WebRTC SDP offers/answers/ICE candidates through clasp.set() on signal addresses. That exact pattern scales to video. The scene graph (layouts, overlays, banners, who's on stage) lives as CLASP state, subscribed to by the compositor.

LVQR replaces RTMP fan-out with MoQ/QUIC. Zero-copy fan-out to N viewers. WebSocket fallback for browsers without WebTransport. RTMP egress when you need YouTube/Twitch compatibility.

Three projects you built. One studio.

Why Client-Side Compositing Works Here

The obvious question: "Isn't server-side compositing more reliable?" Yes, for a SaaS product serving thousands of concurrent broadcasts. But for an open-source self-hosted tool, client-side compositing has massive advantages:

  • No GPU servers needed. The host's browser GPU does the compositing. A modern laptop handles 1080p30 PixiJS rendering without breaking a sweat. LumenCanvas already proves this with projection mapping, which is more demanding than a grid of video feeds.
  • The compositor IS the preview. What the host sees in the studio IS the broadcast output. No "preview vs actual" desync. The PixiJS canvas is both the WYSIWYG editor and the final output.
  • WebCodecs is production-ready. The LVQR test-app already demonstrates the pipeline: MediaStreamTrackProcessor reads frames, VideoEncoder encodes H.264, binary frames go over WebSocket to LVQR. This exact pipeline works with canvas.captureStream() as the source instead of getUserMedia().
  • Zero server-side media processing. LVQR's relay is just passing bytes through. ref-counted, zero-copy fan-out. The server cost is negligible.
Tradeoff
Client-side compositing means the host needs a capable machine and stable upload bandwidth (~6 Mbps for 1080p30). If the host's laptop chokes, the broadcast degrades. For a HeatSync Labs stream or a LumenCanvas demo, this is fine. For enterprise SaaS, you'd want the option to offload to a server-side compositor. The scene graph abstraction makes this swap possible later without changing anything else.

03CLASP as the Scene Graph

CLASP's address-based state model maps naturally to a broadcast scene graph. Every element on screen is a CLASP address. The compositor subscribes to the whole tree. The studio UI writes to it. Plugins read and write to it.

// The entire broadcast state lives in CLASP // Host studio UI writes these via clasp.set() // PixiJS compositor subscribes via clasp.on() // Layout /studio/abc/scene/layout "grid-2x2" /studio/abc/scene/transition { type: "crossfade", ms: 300 } /studio/abc/scene/background { type: "image", src: "/bg.jpg" } // Participants (presence + staging) /studio/abc/participants/p1/name "Moheeb" /studio/abc/participants/p1/stage "live" // "backstage" | "live" /studio/abc/participants/p1/slot 0 /studio/abc/participants/p1/muted false /studio/abc/participants/p2/name "Guest" /studio/abc/participants/p2/stage "backstage" // WebRTC signaling (same pattern as AudioSpace) /studio/abc/signal/p2/p1 { type: "offer", sdp: "..." } /studio/abc/signal/p1/p2 { type: "answer", sdp: "..." } // Overlays /studio/abc/overlays/logo/src "/logo.png" /studio/abc/overlays/logo/pos "top-right" /studio/abc/overlays/logo/visible true /studio/abc/overlays/lower/text "Moheeb Zara — LumenCanvas" /studio/abc/overlays/lower/style "banner-teal" /studio/abc/overlays/lower/visible true // Speaking indicators (stream signals, like AudioSpace) /studio/abc/speaking/p1 { speaking: true, volume: 0.72 } // clasp.stream() // Chat /studio/abc/chat/msg-42 { from: "viewer", text: "love it!", platform: "youtube" } /studio/abc/chat/featured "msg-42" // set to show on screen // Broadcast output state /studio/abc/broadcast/live true /studio/abc/broadcast/recording true /studio/abc/destinations/yt/key "xxxx-xxxx-xxxx" /studio/abc/destinations/yt/status "streaming"
Why CLASP and not just WebSockets
Plain WebSockets give you a message pipe. CLASP gives you stateful addresses with subscriptions, wildcard patterns, atomic bundles, and time-synchronized scheduling. When the host clicks "switch to solo layout," that's a single clasp.set() that instantly propagates to the compositor, every guest's UI, and any plugins subscribed to /studio/*/scene/**. No custom pub/sub infrastructure to build. No state reconciliation logic. CLASP's revision-based conflict resolution handles simultaneous edits (multiple producers). The journal gives you undo and scene recall for free.

Scenes as CLASP Bundles

A "scene" (saved preset) is just a CLASP bundle -- an atomic set of changes applied together:

// Save current scene const sceneSnapshot = await clasp.snapshot('/studio/abc/scene/**'); await clasp.set('/studio/abc/presets/interview', sceneSnapshot); // Recall scene (atomic transition) const preset = await clasp.get('/studio/abc/presets/interview'); clasp.bundle(preset.map(([addr, val]) => ({ set: [addr, val] })));

The compositor sees a burst of scene graph changes arrive atomically. The transition system (crossfade, cut, wipe) is handled by the PixiJS renderer, triggered by the transition config at /studio/abc/scene/transition.

04PixiJS Compositor

The compositor is a vue3-pixi application that renders the broadcast frame. It's conceptually identical to how LumenCanvas renders projection mapping layers -- except the "layers" are video feeds, overlays, and text instead of image effects.

Compositor Architecture

// Simplified compositor structure (vue3-pixi) // Each video feed becomes a PixiJS VideoTexture // Layout engine positions them according to scene graph <template> <Application :width="1920" :height="1080" ref="app"> <!-- Background layer --> <Sprite v-if="scene.background" :texture="bgTexture" /> <!-- Participant video feeds (positioned by layout engine) --> <Container v-for="p in liveParticipants" :key="p.id" :x="layout.slots[p.slot].x" :y="layout.slots[p.slot].y"> <!-- Rounded rectangle mask for camera shape --> <Graphics @render="drawMask(p)" /> <!-- Video texture from WebRTC MediaStream --> <Sprite :texture="videoTextures[p.id]" :width="layout.slots[p.slot].w" :height="layout.slots[p.slot].h" :mask="masks[p.id]" /> <!-- Name plate --> <Text :text="p.name" :style="nameStyle" :x="layout.slots[p.slot].w / 2" :y="layout.slots[p.slot].h - 32" /> <!-- Speaking ring (animated via CLASP stream signal) --> <Graphics v-if="speaking[p.id]" @render="drawSpeakingRing(p)" /> </Container> <!-- Overlay layers --> <Sprite v-for="o in visibleOverlays" :key="o.id" :texture="overlayTextures[o.id]" :x="overlayPos(o).x" :y="overlayPos(o).y" /> <!-- Lower third / banner --> <BannerOverlay v-if="scene.overlays?.lower?.visible" :text="scene.overlays.lower.text" :style-name="scene.overlays.lower.style" /> <!-- Featured chat comment --> <ChatBubble v-if="featuredChat" :message="featuredChat" /> </Application> </template>

Video Textures from WebRTC Streams

When a guest's WebRTC PeerConnection fires ontrack, the MediaStream gets turned into a PixiJS VideoTexture:

// When WebRTC track arrives (signaled through CLASP) pc.ontrack = (ev) => { const videoEl = document.createElement('video'); videoEl.srcObject = ev.streams[0]; videoEl.autoplay = true; videoEl.playsInline = true; videoEl.muted = true; // audio handled separately await videoEl.play(); // Create PixiJS texture from the video element videoTextures[participantId] = PIXI.Texture.from(videoEl); // PixiJS auto-updates the texture each frame };

Canvas Capture to LVQR

The compositor's canvas output gets captured and sent to LVQR using the same pipeline the LVQR test-app already uses:

// Capture the PixiJS canvas as a MediaStream const canvas = pixiApp.canvas; // PixiJS v8 .canvas property const stream = canvas.captureStream(30); // 30fps // Mix audio from all WebRTC participants const audioCtx = new AudioContext(); const destination = audioCtx.createMediaStreamDestination(); for (const peer of Object.values(peers)) { const source = audioCtx.createMediaStreamSource(peer.stream); source.connect(destination); } // Add audio track to the canvas stream destination.stream.getAudioTracks().forEach(t => stream.addTrack(t)); // Encode with WebCodecs (same pattern as LVQR test-app) const videoTrack = stream.getVideoTracks()[0]; const processor = new MediaStreamTrackProcessor({ track: videoTrack }); const reader = processor.readable.getReader(); const encoder = new VideoEncoder({ output: (chunk, meta) => { // Send to LVQR ingest over WebSocket // Wire format: [u8 type][u32 ts][payload] sendToLvqr(chunk, meta); }, error: (e) => console.error(e) }); encoder.configure({ codec: 'avc1.42C028', width: 1920, height: 1080, bitrate: 4_500_000, framerate: 30, latencyMode: 'realtime', avc: { format: 'avc' } }); // Feed frames (runs in a loop) while (broadcasting) { const { value: frame } = await reader.read(); if (frame) { encoder.encode(frame, { keyFrame: frameCount++ % 60 === 0 }); frame.close(); } }
LumenCanvas Precedent
This is the same rendering architecture as LumenCanvas. PixiJS v8 with vue3-pixi, scene graph driven, GPU-accelerated. The only new piece is capturing the canvas output and encoding it, which is a well-documented WebCodecs pattern that LVQR's test-app already implements. The compositor is essentially a specialized LumenCanvas "show" where the layers are live video feeds instead of images and effects.

05LVQR Broadcast Output

The composited H.264 stream enters LVQR through the existing WebSocket ingest endpoint (/ingest/{broadcast}). From there, LVQR handles everything:

  • Remuxing: Incoming H.264 AVCC frames get wrapped in fMP4 (CMAF) segments by lvqr-ingest. This already works.
  • MoQ Fan-Out: The MoQ relay distributes fMP4 segments to all subscribers via QUIC/WebTransport. Zero-copy ref-counted buffers. Adding a viewer costs a refcount increment, not a data copy.
  • WebSocket Fallback: Browsers without WebTransport get fMP4 frames over WebSocket via /ws/{broadcast}. Already implemented.
  • Recording: lvqr-record subscribes to the broadcast and writes fMP4 to disk. Already works.
  • Viewer Playback: The <lvqr-player> web component handles MSE playback. Drop it in a page and point it at the relay.

What LVQR Needs for Full StreamYard Parity

From the LVQR roadmap, the Tier 2 features that matter for this project:

  • RTMP Egress (Tier 2.1 on roadmap): Re-stream the MoQ broadcast to YouTube/Twitch/Facebook RTMP endpoints. This is the biggest gap for StreamYard parity. Without it, viewers can only watch through the LVQR player or WebSocket endpoint.
  • Audio Track Support: Currently the test-app only streams video. AAC encoding via WebCodecs is not yet widely supported. Workaround: encode audio separately and push as a second track, or use MediaRecorder for the audio track and WebCodecs for video.
  • Multi-Codec: HEVC, VP9, AV1 support (Tier 2.2). Not critical for launch but nice for efficiency.
  • LL-HLS Egress: For embedding on websites where you want broader compatibility than WebTransport.
Realistic Launch Path
LVQR can serve viewers directly today (MoQ + WebSocket fallback). RTMP egress to YouTube/Twitch is the priority Tier 2 feature. Until that ships, you could add a lightweight FFmpeg sidecar that subscribes to the LVQR WebSocket output and re-streams to RTMP destinations. Not elegant, but functional as a bridge.

07Scaling Considerations

Participant Limit

WebRTC P2P mesh (as used in AudioSpace) works for up to ~8 participants with video, ~15 with audio only. Each participant maintains a PeerConnection to every other participant. For a live production studio where you typically have 2-6 guests on screen, this is fine. If you need more, you'd add an SFU (LiveKit or mediasoup) as the media transport while keeping CLASP for signaling and state. The CLASP-based signaling pattern doesn't change, you just point the PeerConnections at the SFU instead of P2P.

Viewer Scale

Viewer scale is entirely LVQR's problem, not the compositor's. LVQR's zero-copy fan-out means a single relay can serve hundreds of concurrent viewers. The mesh topology planner (lvqr-mesh) can distribute viewer load to peer relays for larger audiences. And RTMP egress offloads to YouTube/Twitch's CDN for massive scale.

09Plugin Architecture (via CLASP)

CLASP's address-based model makes plugins natural. A plugin is anything that reads or writes to the scene graph. No plugin API to design. No IPC protocol. Just CLASP addresses with capability scoping.

// Auto-Director plugin (switches to active speaker) // This is just a CLASP client that subscribes and writes const clasp = new ClaspSDK('wss://relay.clasp.to'); await clasp.connect({ token: pluginToken }); // scoped to scene:write + speaking:read let currentSpeaker = null; clasp.on('/studio/*/speaking/*', (data, addr) => { const pid = addr.split('/').pop(); if (data.speaking && data.volume > 0.5 && pid !== currentSpeaker) { currentSpeaker = pid; clasp.set('/studio/abc/scene/layout', 'solo'); clasp.set('/studio/abc/scene/solo-target', pid); } }, { maxRate: 10 });

Plugins can be:

  • Client-side (browser): IFrame panel in the studio UI that connects to CLASP and writes to overlay addresses. Custom lower third designer, poll widget, countdown timer.
  • Server-side (any language): Any process that connects to the CLASP relay. Python script, Rust daemon, Node service. Chat aggregator that polls YouTube API and writes to /studio/*/chat/**. AI captioner that reads audio (via a separate WebRTC connection) and writes to /studio/*/overlays/captions.
  • Hardware: CLASP already bridges to MIDI, OSC, DMX. A MIDI controller could switch scenes. An Arduino could trigger overlays. A lighting console could sync colors to the broadcast theme. This is where CLASP's heritage as a creative tooling protocol really shines.
CLASP Capabilities for Plugin Isolation
Each plugin gets a capability token (JWT) scoped to specific address patterns. The auto-director plugin gets read: ["/studio/*/speaking/**"] and write: ["/studio/*/scene/**"]. It can't read chat, it can't modify overlays, it can't see other studios. CLASP enforces this at the relay level. No plugin sandbox runtime needed.

10Why This Architecture

This isn't a hypothetical "what if we used technology X." Every piece of this stack exists and has been tested:

  • CLASP WebRTC signaling works in AudioSpace with real users in real rooms
  • CLASP state subscriptions drive real-time UIs in the chat app, the FPS game, the Crokinole game
  • LVQR WebCodecs-to-relay pipeline works in the test-app with real video from real cameras
  • PixiJS video textures are a core PixiJS feature, and vue3-pixi rendering is LumenCanvas's entire foundation
  • canvas.captureStream() is a standard Web API supported in all modern browsers

The new code is the studio UI (frontend), the layout engine (geometry), and the glue connecting these proven pieces. No new protocols. No new servers. No new infrastructure patterns. Just composition of things that already work.

And because it's all your stack, you can instrument the whole thing. CLASP's signal inspector shows every scene graph change in real time. LVQR's admin API shows every active broadcast and viewer. PixiJS's renderer stats show frame timing. Full observability, no black boxes.
